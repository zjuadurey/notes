| 方向线 | 判断 | 理由 | 这条线大概要做什么 / 可能贡献 |
|---|---|---|---|
| 泛泛做“不同 qLDPC/BB decoder 的速度–精度权衡比较” | **已拥挤，不建议押主线** | QUITS 已经把 qLDPC code family、decoder、noise model 做成模块化 benchmark，并明确报告了 decoding runtime 与 logical failure rate 的 trade-off，还支持 balanced product codes。再做一篇“我们也发现不同 decoder 有权衡”的论文，新意不够。  | 主要会变成：复现若干 decoder，在不同 code / noise 下画 accuracy–latency 曲线。贡献大概率只是“补更多图和更多点”，很难形成新的核心方法或机制。 |
| “Budget-aware / latency-aware decoder selection” 或“根据置信度在 decoder 间切换” | **已很接近现有工作，不建议押主线** | *Decoder Switching* 已经把“弱 decoder 先跑，低置信度时切强 decoder”做成主线，目标就是在接近强 decoder 精度的同时维持接近弱 decoder 的平均速度；作者还明确提到可扩展到 qLDPC/BB code。你现在再做 selection/switching，容易被看成同一路数的变体。  | 主要会变成：设计一个规则或轻量模型，在不同 decoder 之间切换。贡献可能只是“把已有 switching 思路搬到 BB/qLDPC 上”，创新空间偏小。 |
| “再造一个更快的通用 qLDPC decoder” | **竞争太强，不适合现在硬挤** | 这条线上近期已经有 AC、BP+OTF 等工作：AC 在 BB code 上报告了相对 BP-OSD 的显著加速，BP+OTF 直接把自己定位成近线性时间 qLDPC decoder。对你们这种刚进入 QEC 的团队，这条线门槛高、同质竞争强。  | 主要会变成：重新设计 decoder 结构、消息传递规则、后处理或复杂度优化。真正有价值的贡献会很强，但实现和理论门槛都高，短期内不稳。 |
| 流式 / 窗口式 qLDPC 解码改进 | **还有空间，优先考虑** | 这条线对象和 baseline 都现成：已有 sliding-window / low-latency qLDPC decoding 工作，BB 小码、circuit-level noise、latency 和 logical error 都是自然评测对象。你们可以在现有窗口框架里改具体机制，而不是自己造 benchmark。QUITS 也专门把 sliding-window inner decoder 做成可替换模块。  | 主要做法：在现有 sliding-window 框架里改机制，比如跨窗口信息复用、warm start、动态 halt、最后窗口处理。可能贡献：在不改大框架的前提下，同时改善 logical error 和 latency，形成一个很扎实的系统/算法小闭环。 |
| effective-distance gap 的机理分析 + 轻量修复 | **还有空间，优先考虑** | QUITS 直接指出：effective distance 的下降可能来自 sliding-window BP 中的 trapping sets，或 syndrome extraction 中的 hook errors；并明确把“分析 failure mechanism、改进 BP / detector graph、缓解 error floor”列为后续方向。这个题的外部锚点很强，指标也天然。  | 主要做法：先定位哪些 failure pattern 导致 effective distance 掉落，再设计一个轻量修复，例如局部规则、图重构或针对性后处理。可能贡献：解释“为什么现有 decoder 在 circuit-level noise 下失效”，并给出小而有效的补丁。 |
| syndrome extraction circuit / detector graph 与 decoder 协同设计 | **有空间，但比前两条更重** | QUITS 还明确说：对 QLP/BPC，需要能 suppress hook errors、尽量保住 effective distance 的 syndrome-extraction design principle，而且他们的框架正好能测。这条线比“换 decoder”更系统，但实现和实验负担更高。  | 主要做法：调整 syndrome extraction 电路、detector graph 或测量调度，再和 decoder 联合评估。可能贡献：提出一套“电路侧 + 解码侧”协同优化原则，而不只是单独换 decoder。 |
