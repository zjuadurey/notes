# 科研选题策略更新

## 过去的教训

之前在脉冲级量子程序形式化验证上的挫败，不应被理解为“我不适合科研”，而应理解为：

**我在一个需要同时自造对象、平台、基准、评价框架和叙事合法性的题上投入过深。**

这类题的问题不只是难，而是：

- 外部锚点弱
- 证据链脆弱
- 问题成立依赖大量自定义前提
- 一旦结果不强，整篇工作很容易失去支撑

所以，我以后不应再优先押这种题。

---

## 我当前更适合的题型

我更适合优先做这类题：

- 已有真实对象
- 社区承认这个对象本身值得研究
- baseline 可以很快列出
- 指标天然成立
- 最小闭环近
- 即使结果一般，也能留下经验规律、负结果 insight 或 benchmark/harness 价值

更具体地说，我更适合：

- 已有实体系统上的资源权衡
- 选择 / 调度 / 配置策略
- benchmark / harness / empirical characterization
- 基于真实 workload 的系统经验规律提炼

而不适合优先做：

- 语言 / IR / 语义层，但缺少真实 workload 的题
- 需要自己发明对象和评价体系的题
- 新意主要靠 framing、而不是靠外部对象站住的题
- 看起来很高级，但实验闭环很远的题

---

## 什么是健康的题

一个健康的题，不一定看起来最前沿、最高级，但它通常有一个很重要的特征：

**越讨论越清楚，而不是越讨论越奇怪。**

也就是说，随着推进，这个题会越来越容易回答下面这些问题：

- 对象是什么
- 冲突是什么
- baseline 是什么
- 指标是什么
- 最小闭环怎么跑
- 结果出来后能说明什么

健康的题通常有以下特征：

### 1. 对象是硬的

不需要我反复解释“这到底算不算一个独立问题”。

对象本身就是社区承认的，例如：

- decoder
- workload
- trace
- benchmark
- kernel
- scheduler target

### 2. 冲突是自然的

系统矛盾不是我自己定义出来的，而是现实里天然存在的，例如：

- latency vs throughput
- accuracy vs cost
- memory vs performance
- utilization vs fairness
- tail latency vs average efficiency

### 3. framing 是在澄清问题，不是在支撑问题

健康的题当然也需要 framing，但 framing 的作用应该是：

**把一个本来就存在的问题层级讲清楚。**

而不是：

**靠 framing 才勉强把这个题撑成一个独立问题。**

换句话说：

- 好的 framing 是解剖刀
- 坏的 framing 是支架

### 4. 边界是稳定的

一个健康的题不会在讨论中不断滑向别的层级。

不会出现这种情况：

- 往下说，就只是一个普通调度问题
- 往上说，就变成一个过大的平台愿景
- 停在中间，又必须不断解释为什么这一层值得单独研究

如果一个题总在相邻层级之间来回滑，说明它的边界不够稳。

### 5. 负结果也能留下东西

健康的题即使结果一般，也能留下：

- insight
- tradeoff characterization
- empirical law
- benchmark / harness 价值
- failure boundary

而不是一旦没赢 baseline，整篇就失去意义。

## 以后选题前的六个门槛

以后任何题，先过这六个门槛。  
只要有两个门槛明显过不了，就先不做。

### 1. 测试对象是不是独立于我而存在

不用我自己发明，社区里本来就承认它是个对象。

例如：

- decoder
- code family
- public traces
- standard kernels
- real workloads
- benchmark suites

如果对象本身都需要我先论证“它到底算不算一个合理研究对象”，这个题就已经危险。

### 2. baseline 能不能在一周内列出来

注意不是做出来，而是列出来。

如果一周后我还说不清：

- 该和谁比
- 为什么和它比
- 它们分别代表什么思路

那通常说明题目站不稳。

### 3. 指标是不是天然成立

最好是别人一看就懂，不需要我花很多篇幅解释“为什么这个指标重要”。

优先选择这种指标：

- latency
- throughput
- error rate
- memory
- energy
- utilization
- slowdown
- deadline miss rate

如果一个题需要我先发明一套只有自己懂的指标体系，风险很大。

### 4. 最小闭环能否在 3 到 5 天内跑通

不是完整论文，而是一个最小图：

- 两种方法
- 两个对象
- 一个核心结论图

如果这个最小闭环都想不出来，就先别投入。

### 5. 负结果有没有价值

一个健康的题即使结果不惊艳，也能留下：

- 稳定 insight
- empirical law
- benchmark / harness 价值
- tradeoff characterization
- failure boundary

如果结果一旦不强，整篇就会塌掉，那说明题本身太依赖“赢一次”。

### 6. 问题成立是不是主要靠 framing

这是我现在新增的一条。

有些题表面上看很新，但仔细想会发现：

- 对象不是新的
- 瓶颈不是新的
- 方法也不是新的
- 真正的新意主要来自“换了一个更高级的说法”

这类题不是不能做，但对我当前阶段不够健康。

以后我要警惕这种情况：

- 如果把题目措辞降一层，它就变成普通调度 / 普通系统优化
- 如果把题目措辞升一层，它又变成更大的平台愿景
- 中间那层所谓“新问题”主要靠语言包装维持

这类题的创新点往往太脆弱，不适合我当前优先押主线。

## 一个新的排雷问题：我要自己发明几层东西？

以后每个题都问自己一句：

**为了让这个题成立，我需要自己发明几层东西？**

如果需要同时发明两层以上，就要高度警惕。

常见层包括：

1. 对象层：研究对象本身
2. 接口层：系统边界 / 抽象
3. workload 层：输入数据或任务集
4. baseline 层：比较对象
5. metric 层：评价方式
6. story 层：为什么这是一个独立问题

理想情况是：

- 对象层现成
- workload 层现成或容易借用
- metric 层天然成立
- baseline 层很快能列出
- 我主要只需要在“方法层”创新

如果一个题需要我同时补对象、workload、metric、story 四层，那基本不该做。

---

## 一个新的判断标准：题是不是越讨论越健康

以后看一个题时，除了看对象、baseline、指标和最小闭环，我还要额外问自己：

**这个题是越讨论越清楚，还是越讨论越奇怪？**

如果一个题越推进越能明确：

- 它的对象
- 它的边界
- 它的 baseline
- 它的指标
- 它的最小实验

那它通常是健康的题。

如果一个题越推进越需要不断解释：

- 为什么这是个独立问题
- 为什么不是已有问题的换皮
- 为什么这个层级值得单独成立
- 为什么这些设定不是我自己硬定义出来的

那它通常不是我当前阶段应该押的主线。

---

## 我当前的选题优先级原则

以后选题时，优先级按下面排序：

### 第一优先级

外部锚点强、最小闭环近、负结果也有价值、并且越讨论越清楚的题。

### 第二优先级

对象和指标都稳，baseline 清楚，方法空间还需要细化的题。

### 第三优先级

概念上新，但需要较多 framing 才能站住的题。

### 最后优先级

需要我自己同时发明对象、平台、基准和评价框架的题。

---

## 给自己的提醒

- 不要再把“痛苦”误判成“有研究价值”。
- 不要把“概念上高级”误判成“题目健康”。
- 不要因为一个题能讲出很漂亮的故事，就忽视它是否真的有独立对象和自然证据链。
- 不要把 framing 当成支架，只把它当成解剖刀。
- 如果一个题越讨论越需要解释它为什么成立，就不要把它作为当前主线。
- 选题时优先找外部锚点强、最小闭环近、即使负结果也能留下 insight 的方向。
- 先做能形成闭环的题，再考虑更抽象、更底层、更宏大的叙事。
- 当前阶段，**宁可做一个硬一点、小一点、站得住、越讨论越清楚的题，也不要做一个高级但需要靠包装维持的新题。**
