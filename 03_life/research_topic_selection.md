# 科研选题反思

## 脉冲级量子程序的形式化验证：这个选题的问题

上次做脉冲级量子程序的形式化验证时，遇到的核心问题不是单纯“执行得不够好”，而是选题机制本身有缺陷。

### 1. 生态断裂

- Qiskit 已经抛弃了 pulse 叙事。
- 很难找到像 Qiskit 代码那样的真实 pulse 程序作为测试对象，只能自己构造。
- pulse 层只能自己写一层中间实现，成熟度和可信度都不强。
- 没有现成的社区 benchmark、真实 workload、标准接口和稳定工具链。

### 2. 工作闭环不自然

一个健康的题通常是：

已有对象 -> 定义问题 -> 设计方法 -> 用公共或公认数据评测

上次那个题实际变成了：

自己造语言/中间层 -> 自己造程序 -> 自己定义正确性 -> 自己找指标

这会导致整篇工作的证据链很脆弱，也很容易让我自己在推进过程中持续心虚。

### 3. 缺少外部锚点

- 没有现成 baseline。
- 没有公认 workload。
- 没有大家熟悉的 evaluation target。

结果就是：

- 不知道实验应该怎么设计。
- 不知道做到什么程度算够。
- 很容易变成“玩具系统证明玩具性质”。

## 这次经历带来的判断

这次挫败不应该被理解成“我不会做科研”，更准确的结论是：

- 我当前不适合优先做“语言/IR/形式化语义层，但缺乏真实 workload 和成熟生态”的题。
- 我需要主动避开那些要求我自己同时发明对象、平台、基准和评价框架的题。
- 我更适合做“已有实体系统上的资源权衡、选择策略、benchmark/harness、经验规律提炼”这类题。

这类题的外部锚点更强，心理负担更小，也更容易在有限时间内收敛。

## 以后选题前的五个门槛

以后任何题先过这五个门槛。只要有两个门槛明显过不了，就先不做。

### 1. 测试对象是不是独立于我而存在

也就是不用我自己发明，社区里本来就承认它是个对象。

例如：

- code
- decoder
- public traces
- standard kernels

### 2. baseline 能不能在一周内列出来

注意不是做出来，而是列出来。

如果连 baseline 名单都很难写，通常说明题目站不稳。

### 3. 指标是不是天然成立

最好是别人一看就懂，不需要额外花很多篇幅解释为什么这个指标重要。

比较好的指标包括：

- latency
- throughput
- error rate
- memory
- energy

### 4. 最小闭环能否在 3 到 5 天内跑通

不是完整论文，而是一个最小图：

- 两种方法
- 两个对象
- 一个核心结论图

如果这个闭环都想不出来，先别投入时间。

### 5. 负结果有没有价值

一个好题即便结果不惊艳，也能留下可写的 insight。

如果结果一旦不强，整篇就会塌掉，那通常说明题本身不够健康。

## 给自己的提醒

- 不要再把“痛苦”误判成“有研究价值”。
- 不要因为题目看起来高级，就忽视对象、baseline 和评测是否站得住。
- 选题时优先找外部锚点强、最小闭环近、即使负结果也能留下 insight 的方向。
- 先做能形成闭环的题，再考虑更抽象、更底层、更宏大的叙事。
